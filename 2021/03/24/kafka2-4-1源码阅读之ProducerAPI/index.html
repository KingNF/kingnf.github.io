kafka2.4.1 源码阅读

Producer API

本文章仅适合对kafka 有基础的同学。

思考两个问题：

  1. 当kafka client 版本小于 kafka broker 版本时，且两个版本具有不同的消息magic verison版本时，kafka 的producer API是如何处理的？
  2. 当kafka client 版本大于 kafka broker 版本时，且两个版本具有不同的消息magic verison版本时，kafka 的producer API是如何处理的？

带着这两个问题，我们从kafka 2.4.1的源码出发，判断是否能够从中找到答案。

预备知识

针对Producer API 首先我们必须要知道，Producer API request Version 的含义，这个request Version 是 producer 向kafka broker 发送请求消息的类型版本，以及格式，截止kafka2.4.1中，kafka broker 总共支持8个Request API 版本，其中有几个关键的Producer API 的Version 是值得注意的。

  • V0版本开始，支持messge format V0版本，同时支持 ack 请求参数的设置，0 不确认，1只有leader确认，-1 全部ISR确认。
  • V2版本开始，支持 message format V1版，message format 中存在 timestamp 和 relative offest 。 kafka相关版本（0.10.0 以上）
  • V3版本开始，支持message format V2版，并且同时支持事务，和幂等特性。kafka版本（0.11.0 以上）

  • V5版本开始，partition level 开始支持the log_start_offset字段。

  • V7版本开始，支持ZStandard 压缩格式，具体见KIP-110。

  • 从segment的第一个时间以来的最大时间戳已经过期，（如果第一条消息没有时间戳，则从segment 的创建时间开始）
  • index 满了

至此，我们已经基本将producer 处理API的请求的基本逻辑理清楚了，下面让我们来回答一下，文章开头提出的两个问题：

  1. 当kafka client 版本小于 kafka broker 版本时，且两个版本具有不同的消息magic verison版本时，kafka 的producer API是如何处理的？

    答： 当client 版本小于broker 版本时，对于生产者来说，消息的格式，是在producer那里就完成了消息格式的封装，我们为了减少问题的难度，先不考虑压缩问题，当producer 将封装好的消息，发送到broker端的时候，broker端可以正常对client请求进行解析，一般高版本broker对client都有很好的兼容性，broker端，会按照client端支持的API版本与client通信，然后根据API request Version将消息转换为MemoryRecord 对象，之后，在将消息存入broker服务器的时候，broker如果检测到producer中消息的格式和broker中消息的格式不同的时候，会将低版本消息通过MemroyRecordBuilder 转换为broker 服务器端对应的存储方式，在这过程中会进行Record和Batch的检验，丢弃不完整的消息等，同时有个点需要注意：不管有没有消息版本的转换，或者消息的压缩，都会对producer发送来的消息，进行解析，然后给每条消息分配offset，统计每个batch最大的offset，最大时间戳等信息。只不过消息转换的时候，需要借助MemoryBuilder 根据broker message format 类型，重新对消息的值进行封装，如果低版本中缺少高版本值的字段的话，高版本对应的field，会用默认值代替。（produceid 用-1，produce epoch 用-1，baseSequence 用-1.isTransactional 用 false 代替）。

  2. 当kafka client 版本大于 kafka broker 版本时，且两个版本具有不同的消息magic（messag format） verison版本时，kafka 的producer API是如何处理的？

    答：这个需要两种情况进行讨论，第一种情况是：broker 服务器版本就是比 client 服务器版本低的情况。第二种是：broker服务器版本和client服务器版本一样，但是broker服务器的topic对应设置的messag format version 版本比较低。

    • 针对第一种情况：当client 发送给broker 较高的API request version的时候，broker接收到client 的 producer request 请求，发现API request Version 无法解析，则会给client 返回 request v0版本的消息，来确保client 可以解析出v0版本的response 格式，这时，则就要看client 有没有相关的处理逻辑，如果存在的话，则会利用v0版本的API和broker 沟通出，brokr所能支持的最大的Producer Request API version，然后以该API version 已经对应的messag format version与broekr kafka 进行通信，来完成消息的生产。
    • 针对第二种情况：client 以高版本 produce request API和对应的高版本 message format version 来访问对应的broker 服务器的时候，服务器是可以正常完成，对produce request 的解析的，也可以正常对produce data 数据进行解析为MemoryRecord的时候。此时broker 服务器存储消息的时候，同样会利用MemoryRecordBuilder 将 高版本 message format version 转换为低版本message format version，此时会讲高版本特有的字段给忽略掉。
  • 注意： 当一个ProduceRequest中包含多个Batch的时候（v2 magic version的时候，会报错，V0和V1不会），broker端在做消息转换的时候，会将一个request中的 多个batch 转换为一个batch。
